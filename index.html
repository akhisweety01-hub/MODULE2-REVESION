Module 01

Imbalanced Data & Advanced Classification

When the minority class is the one that matters — fraud, churn, disease — accuracy lies to you. This module covers resampling theory, focal loss, and the metrics that actually reflect performance under skew.

Why Accuracy Fails

With a 99:1 class ratio, a classifier that always predicts the majority class scores 99% accuracy while catching zero true positives. The fix is to evaluate on metrics sensitive to the minority class, and to address the imbalance at the data or loss-function level.

SMOTE — Synthetic Minority Oversampling

SMOTE generates synthetic minority samples by interpolating between a real point and one of its k-nearest minority neighbors, rather than simply duplicating rows (which causes overfitting).

$$x_{new} = x_i + \lambda \cdot (x_{zi} - x_i), \quad \lambda \sim U(0,1)$$

where $x_i$ is a minority sample and $x_{zi}$ is one of its $k$ nearest minority-class neighbors.

ADASYN (Adaptive Synthetic Sampling) extends this by generating more synthetic points for minority samples that are harder to learn — those with more majority-class neighbors — weighting density by:

$$\Gamma_i = \frac{\Delta_i / k}{\sum_j \Delta_j/k}, \qquad \Delta_i = \text{majority-class neighbors of } x_i$$
python · smote_pipeline.py
from imblearn.over_sampling import SMOTE, ADASYN
from sklearn.model_selection import train_test_split

X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)

# Resample ONLY the training fold — never touch the test set
smote = SMOTE(sampling_strategy='auto', k_neighbors=5, random_state=42)
X_res, y_res = smote.fit_resample(X_train, y_train)

# ADASYN focuses synthetic points near the decision boundary
adasyn = ADASYN(sampling_strategy='auto', n_neighbors=5, random_state=42)
X_res2, y_res2 = adasyn.fit_resample(X_train, y_train)

Focal Loss

Focal Loss, from RetinaNet, down-weights easy, well-classified examples so the model spends gradient on hard, minority-class examples:

$$FL(p_t) = -\alpha_t (1-p_t)^{\gamma} \log(p_t)$$

Here $p_t$ is the model's predicted probability for the true class, $\gamma \geq 0$ is the focusing parameter (typically 2), and $\alpha_t$ balances class frequency. When $p_t \to 1$ (an easy, correct example), $(1-p_t)^\gamma \to 0$ and the loss contribution vanishes; when $p_t$ is low (a hard example), the modulating term stays near 1 and the full cross-entropy penalty applies.

Precision-Recall vs ROC-AUC

ROC-AUC plots TPR vs FPR across thresholds. It can look deceptively good under imbalance because FPR's denominator (true negatives) is huge.

$$TPR=\frac{TP}{TP+FN}, \quad FPR=\frac{FP}{FP+TN}$$

PR-AUC plots Precision vs Recall, both of which depend only on the positive class — making it the preferred metric for rare-event detection.

$$Precision=\frac{TP}{TP+FP}, \quad Recall=\frac{TP}{TP+FN}$$

Resampling must happen strictly inside the training fold to avoid leaking synthetic information into the test set.

Interview Q

Why not oversample before train/test split?

Tap to reveal
Answer

Synthetic points generated from the full dataset can land in the test fold, meaning the model gets evaluated on samples that share interpolated information with training data — an optimistic, leaky estimate of real-world performance.

Trade-off

SMOTE vs class weights

Tap to reveal
Answer

Class weights are cheaper (no new rows, no distance computation) and work for any model with a loss function. SMOTE physically densifies the minority region, which helps distance-based and tree-split methods that don't use a weighted loss.

Definition

What does $\gamma$ control in Focal Loss?

Tap to reveal
Answer

$\gamma$ is the focusing parameter. Higher $\gamma$ more aggressively suppresses the loss contribution of easy examples, forcing training to concentrate on hard, often minority-class, examples. $\gamma=0$ reduces Focal Loss to plain cross-entropy.

Micro-Quiz

A fraud model has 99.4% accuracy but 0.12 recall on the fraud class. What's the most likely explanation?

High accuracy with low recall on the minority class is the textbook signature of class imbalance — the model can satisfy accuracy almost entirely by getting the majority class right, while contributing little to detecting the rare positive class. Address with resampling, focal loss, or cost-sensitive learning, and re-evaluate on PR-AUC/recall.
Module 02

Feature Engineering & Sklearn Pipelines

The single biggest source of inflated offline metrics is data leakage. Pipelines aren't just convenience — they're a leakage firewall between your training fold and everything downstream.

Data Leakage — The Cardinal Sin

Leakage occurs when information from outside the training set (often, statistics of the test set) influences model training. The classic mistake: fitting a StandardScaler or TargetEncoder on the full dataset before splitting. Even though no labels are technically passed to the test set, its mean/variance leaked into the scaler — the model sees a feature distribution that "knows about" data it hasn't met.

Rule: every transformer that learns statistics from data (mean, std, target-conditional means, vocabulary) must be .fit() only on the training fold, ideally enforced inside a Pipeline so cross-validation refits it correctly for every fold.

ColumnTransformer + Custom Transformers

Real datasets mix numeric, categorical, and text columns that need different treatment. ColumnTransformer applies a different transformer to each named subset of columns, in parallel, and concatenates the outputs.

python · feature_pipeline.py
from sklearn.compose import ColumnTransformer
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.base import BaseEstimator, TransformerMixin

class OutlierCapper(BaseEstimator, TransformerMixin):
    """Custom transformer: caps values beyond 1.5*IQR."""
    def __init__(self, factor=1.5):
        self.factor = factor

    def fit(self, X, y=None):
        q1, q3 = np.percentile(X, [25, 75], axis=0)
        iqr = q3 - q1
        self.lower_ = q1 - self.factor * iqr
        self.upper_ = q3 + self.factor * iqr
        return self

    def transform(self, X):
        return np.clip(X, self.lower_, self.upper_)

numeric_cols = ['income', 'age', 'balance']
categorical_cols = ['job', 'marital']

numeric_pipe = Pipeline([
    ('impute', SimpleImputer(strategy='median')),
    ('cap', OutlierCapper()),
    ('scale', StandardScaler()),
])

preprocessor = ColumnTransformer([
    ('num', numeric_pipe, numeric_cols),
    ('cat', OneHotEncoder(handle_unknown='ignore'), categorical_cols),
])

full_pipeline = Pipeline([
    ('prep', preprocessor),
    ('model', RandomForestClassifier(random_state=42)),
])

Target Encoding — Math & The Leakage Trap

Target encoding replaces a categorical level with the mean of the target for that level, smoothed toward the global mean to avoid overfitting rare categories:

$$\hat{x}_c = \frac{n_c \cdot \bar{y}_c + m \cdot \bar{y}_{global}}{n_c + m}$$

where $n_c$ is the count of category $c$, $\bar{y}_c$ its mean target, $\bar{y}_{global}$ the overall mean, and $m$ a smoothing strength. Danger: computing $\bar{y}_c$ using a row's own label leaks that row's target into its own feature — always use out-of-fold target statistics (e.g. category_encoders.TargetEncoder with cross-validation internals, or a manual K-fold scheme).

Trade-off

Pipeline vs. manual preprocessing

Tap to reveal
Answer

Manual preprocessing (fit scaler once, transform everywhere) silently leaks fold statistics during cross-validation. A Pipeline re-fits every transformer per fold, ensuring the CV score reflects true generalization.

Interview Q

Why handle_unknown='ignore' in OHE?

Tap to reveal
Answer

Production data can contain a category never seen in training (a new city, a new job title). Without handle_unknown='ignore', the encoder raises an error and the pipeline crashes; with it, the unseen category is encoded as all-zeros instead.

Definition

What is $m$ in target encoding smoothing?

Tap to reveal
Answer

$m$ is a pseudo-count that controls how strongly a rare category is pulled toward the global mean rather than trusting its own (noisy, low-sample) mean. Larger $m$ = more smoothing, less overfitting to rare levels.

Evacuation 2 Summary

You fit a StandardScaler on the entire dataset, then split into train/test. What's wrong?

Even unsupervised transforms can leak — the scaler's learned mean and std are now contaminated by test-set values, giving an overly optimistic view of how the pipeline generalizes. Always split first, then fit_transform only on the training fold (a Pipeline + cross-validation enforces this automatically).
Module 03

Git, Version Control & Code Quality for ML

Models are code plus data plus randomness. Reproducibility demands disciplined version control for all three, and clean, linted code so a teammate (or future-you) can trust the pipeline.

Branching Strategy for ML Experiments

A common pattern: main holds production-ready models, develop integrates tested features, and short-lived experiment/* branches isolate each hyperparameter sweep or architecture change — preventing half-finished experiments from polluting the trunk.

bash · git workflow
# Initialize tracking for a new ML project
git init
git add .
git commit -m "Initial commit: project scaffold"

# Isolate a new experiment
git checkout -b experiment-xgboost

# After validating results, merge back
git checkout develop
git merge experiment-xgboost
git branch -d experiment-xgboost

DVC — Versioning What Git Can't

Git is built for text diffs, not multi-gigabyte datasets or binary model checkpoints. DVC (Data Version Control) stores a lightweight pointer (a hash) in Git while the actual data/model file lives in remote storage (S3, GCS, or a local cache) — so git log shows exactly which dataset version produced which model.

bash · dvc basics
dvc init
dvc add data/train.csv         # creates train.csv.dvc pointer
git add data/train.csv.dvc .gitignore
git commit -m "Track training data v1 with DVC"
dvc remote add -d storage s3://my-bucket/dvcstore
dvc push

Code Quality: PEP 8, Black, Pre-commit

PEP 8 is Python's style guide (naming, spacing, line length). Black is an opinionated auto-formatter that enforces a single style with zero configuration debate. Pre-commit hooks run linters/formatters automatically before every commit, so unformatted or broken code never reaches the repo.

Reproducibility: Fixing Randomness

Without a fixed seed, NumPy's RNG, Python's random, and most ML libraries draw different random numbers on every run — making "I got 91% accuracy" unfalsifiable. Seeding every RNG source is the minimum bar for a reproducible experiment.

python · seed_everything.py
import random
import numpy as np

def seed_everything(seed: int = 42):
    random.seed(seed)
    np.random.seed(seed)

seed_everything(42)
Trade-off

Git LFS vs DVC

Answer

Git LFS stores large files but is tied to your Git remote's storage/bandwidth quota. DVC decouples data storage from Git entirely — point it at any remote (S3, GCS, SSH) — and adds first-class ML pipeline/experiment tracking on top.

Why seed both NumPy and Python's random?

Tap to reveal
Answer

They maintain independent RNG states. A library might use either internally (e.g. sklearn often uses NumPy, while some shuffling utilities use Python's random) — seeding only one leaves the other source of nondeterminism unfixed.

Definition

What does a pre-commit hook actually do?

Tap to reveal
Answer

A script Git runs automatically right before finalizing a commit. If the hook (e.g. Black or Flake8) exits with a failure code, the commit is aborted — enforcing style/quality gates without relying on developer discipline alone.

Micro-Quiz

Your teammate's notebook gets 94% accuracy, yours gets 89% on "identical" code and data. What's the most likely root cause?

Unseeded randomness in train/test splitting, model weight initialization, or bootstrap sampling produces different results on every run — even with identical code and data. seed_everything()-style functions, applied before any random operation, are the standard fix.
Module 04

Regularization — Ridge, Lasso & ElasticNet

Regularization trades a little training-set fit for a lot of generalization, by penalizing coefficient magnitude. The choice of penalty shapes which coefficients survive.

Ridge Regression (L2)

Ridge adds the squared $\ell_2$ norm of coefficients to the ordinary least squares loss:

$$\hat{\beta}_{ridge} = \arg\min_\beta \left\| y - X\beta \right\|_2^2 + \lambda \sum_{j=1}^p \beta_j^2$$

This has a closed-form solution: $\hat{\beta} = (X^TX + \lambda I)^{-1}X^Ty$. Adding $\lambda I$ makes $X^TX$ invertible even when features are collinear, which is exactly why Ridge is the standard fix for multicollinearity. Coefficients shrink toward zero but rarely hit it exactly.

Lasso Regression (L1) — Sparsity

$$\hat{\beta}_{lasso} = \arg\min_\beta \left\| y - X\beta \right\|_2^2 + \lambda \sum_{j=1}^p |\beta_j|$$

Unlike L2, the L1 penalty has a non-differentiable kink at zero. Geometrically, the L1 constraint region is a diamond (in 2D) with corners on the axes; the elliptical OLS loss contours are likely to first touch this region exactly at a corner, where one coefficient is precisely zero. This is why Lasso performs automatic feature selection while Ridge does not.

Geometric Intuition

The diamond's sharp corners make zero-coefficient solutions geometrically likely; the circle has no corners, so coefficients shrink but stay non-zero.

ElasticNet — Best of Both

$$\hat{\beta}_{en} = \arg\min_\beta \left\| y - X\beta \right\|_2^2 + \lambda \left[ \alpha \sum_j |\beta_j| + (1-\alpha)\sum_j \beta_j^2 \right]$$

$\alpha \in [0,1]$ mixes the two penalties: $\alpha=1$ is pure Lasso, $\alpha=0$ is pure Ridge. ElasticNet is especially useful when features are correlated in groups — Lasso alone tends to arbitrarily pick one feature from a correlated group and zero out the rest, while ElasticNet's L2 component encourages correlated features to be selected (or shrunk) together.

Both Lasso and ElasticNet are typically solved via coordinate descent: cycle through coefficients one at a time, holding all others fixed, and minimize the (now 1-D, soft-thresholding) sub-problem in closed form — repeating until convergence.

python · regularized_regression.py
from sklearn.linear_model import Ridge, Lasso, ElasticNet
from sklearn.model_selection import GridSearchCV

# Ridge: use when multicollinearity is detected (high VIF)
ridge = Ridge(alpha=1.0, random_state=42)
ridge.fit(X_train_scaled, y_train)

# ElasticNet hyperparameter sweep over both lambda and the L1 ratio
param_grid = {'alpha': [0.01, 0.1, 1, 10], 'l1_ratio': [0.1, 0.5, 0.9]}
grid = GridSearchCV(ElasticNet(random_state=42), param_grid, cv=5,
                     scoring='neg_mean_squared_error')
grid.fit(X_train_scaled, y_train)
Interview Q

Why does Lasso zero out coefficients but Ridge doesn't?

Tap to reveal
Answer

Geometrically, the L1 ball has corners on the axes where one coefficient is exactly zero; the loss contours are likely to intersect there first. The L2 ball is smooth with no corners, so the intersection point generically has all coefficients non-zero, just shrunk.

Trade-off

When would you pick ElasticNet over pure Lasso?

Tap to reveal
Answer

When predictors are highly correlated in groups. Pure Lasso arbitrarily keeps one and drops the rest; ElasticNet's L2 term encourages correlated features to share the shrinkage, giving more stable, interpretable coefficient selection.

Definition

What problem does VIF detect?

Tap to reveal
Answer

Variance Inflation Factor quantifies multicollinearity — how much a predictor's variance is inflated because it's linearly predictable from other predictors. VIF > 5–10 typically signals problematic collinearity, motivating Ridge/ElasticNet over plain OLS.

Micro-Quiz

You have 500 features, most irrelevant, and you suspect only ~10 actually matter. Which regularizer is the most natural starting point?

Lasso's L1 penalty drives many coefficients to exactly zero, performing implicit feature selection — ideal when you believe the true model is sparse. (Note: α=0 in ElasticNet is actually pure Ridge, the opposite of sparse — α=1 is pure Lasso.)
Module 05

Advanced Regression Techniques

OLS assumes Gaussian, homoscedastic errors — an assumption the real world rarely respects. This module covers robust losses and GLMs for when outliers, skew, or count data break that assumption.

Huber Loss — Robust to Outliers

Squared-error loss penalizes large residuals quadratically, so a handful of outliers can dominate the fit. Huber loss behaves quadratically near zero but linearly for large residuals:

$$L_\delta(r) = \begin{cases} \frac{1}{2}r^2 & |r| \leq \delta \\ \delta\left(|r| - \frac{1}{2}\delta\right) & |r| > \delta \end{cases}$$

$\delta$ controls the transition point. Small residuals (inliers) get the smooth, statistically efficient $\ell_2$ treatment; large residuals (outliers) only contribute linearly, capping their influence on the gradient.

Quantile Regression

Rather than estimating the conditional mean, quantile regression estimates a conditional quantile $\tau \in (0,1)$ (e.g. the median, $\tau=0.5$, or the 90th percentile) by minimizing the pinball loss:

$$L_\tau(r) = \begin{cases} \tau \cdot r & r \geq 0 \\ (\tau - 1)\cdot r & r < 0 \end{cases}$$

This asymmetric penalty is invaluable for forecasting ranges — e.g. predicting the 10th and 90th percentile of delivery time gives a usable prediction interval, which a single mean estimate cannot.

GLMs: Poisson & Gamma Regression

OLS assumes targets can be negative and normally distributed — wrong for count data (always non-negative integers) or strictly positive, right-skewed data (e.g. insurance claim amounts). Generalized Linear Models fix this with a link function $g$ connecting the linear predictor to the mean:

$$g(\mathbb{E}[y\,|\,X]) = X\beta$$

Poisson regression (link: $\log$) models count outcomes: $\mathbb{E}[y] = e^{X\beta}$, ensuring predictions stay positive. Gamma regression models strictly positive, right-skewed continuous targets (claim severity, wait times) and naturally handles the heteroscedasticity where variance grows with the mean.

python · robust_glm.py
from sklearn.linear_model import HuberRegressor, QuantileRegressor
from sklearn.linear_model import PoissonRegressor, GammaRegressor

# Robust to outliers in residuals
huber = HuberRegressor(epsilon=1.35)
huber.fit(X_train, y_train)

# Predict the 90th percentile of the target
q90 = QuantileRegressor(quantile=0.9, alpha=0.01, solver='highs')
q90.fit(X_train, y_train)

# Count data (e.g. number of claims)
poisson_model = PoissonRegressor(alpha=1e-3)
poisson_model.fit(X_train, y_counts)
Interview Q

Why is Huber loss "best of both worlds"?

Tap to reveal
Answer

It's quadratic (like MSE) near zero, giving smooth, efficient gradients for typical points, but linear (like MAE) for large residuals, capping the influence any single outlier can exert on the fit.

Trade-off

Mean prediction vs quantile regression

Tap to reveal
Answer

A mean prediction gives one number with no sense of spread. Fitting multiple quantile regressors (e.g. τ=0.1, 0.5, 0.9) yields a full predictive interval — essential for risk-aware decisions like staffing or inventory.

Definition

What is a "link function" in a GLM?

Tap to reveal
Answer

The function $g$ connecting the linear predictor $X\beta$ to the mean of the response distribution, $g(\mathbb{E}[y]) = X\beta$. It lets the model respect constraints (e.g. log-link keeps Poisson predictions strictly positive) that plain linear regression can't.

Micro-Quiz

You're modeling the number of customer support tickets per day (always a non-negative integer). Which model is most appropriate?

Poisson regression's log-link constrains predictions to be positive and its underlying distribution naturally fits non-negative count data, unlike OLS which can predict negative ticket counts. Gamma regression is for continuous, strictly-positive, right-skewed data, not counts.
Module 06

Time Series Forecasting

Time series breaks the i.i.d. assumption underlying most ML — order matters, and the past literally causes the future. Stationarity is the gatekeeping concept for classical forecasting models.

Stationarity & The ADF Test

A series is (weakly) stationary if its mean, variance, and autocovariance don't change over time. Most classical models (ARIMA) require stationarity to produce stable, interpretable parameter estimates.

The Augmented Dickey-Fuller (ADF) test checks for a unit root — the formal signature of non-stationarity. It tests the regression:

$$\Delta y_t = \alpha + \beta t + \gamma y_{t-1} + \sum_{i=1}^p \delta_i \Delta y_{t-i} + \varepsilon_t$$

Null hypothesis $H_0: \gamma = 0$ (a unit root exists → non-stationary). A small p-value ($<0.05$) rejects $H_0$, supporting stationarity. If the series is non-stationary, differencing ($y_t - y_{t-1}$) usually fixes it.

python · stationarity_check.py
from statsmodels.tsa.stattools import adfuller

result = adfuller(series.dropna())
print(f"ADF Statistic: {result[0]:.4f}")
print(f"p-value: {result[1]:.4f}")

# Make non-stationary series stationary via differencing
series_diff = series.diff().dropna()

ARIMA / SARIMAX — The Math

ARIMA($p,d,q$) combines three components: AR($p$) — regression on the series' own past values; I($d$) — differencing $d$ times to induce stationarity; MA($q$) — regression on past forecast errors:

$$\underbrace{\left(1 - \sum_{i=1}^p \phi_i L^i\right)}_{AR} (1-L)^d y_t = \underbrace{\left(1 + \sum_{j=1}^q \theta_j L^j\right)}_{MA} \varepsilon_t$$

where $L$ is the lag operator ($Ly_t = y_{t-1}$). SARIMAX extends this with seasonal AR/MA terms at a fixed period $s$ (e.g. 12 for monthly data with yearly seasonality) and optional exogenous regressors $X$.

Exponential Smoothing & Rolling Features

Simple exponential smoothing weights recent observations more heavily, with weight decaying geometrically:

$$\hat{y}_{t+1} = \alpha y_t + (1-\alpha)\hat{y}_t, \quad \alpha \in (0,1)$$

Holt-Winters extends this with trend and seasonal components. For tree-based forecasters (XGBoost on tabular time data), rolling window features manually encode temporal structure: rolling mean, rolling std, lag-1, lag-7, etc., feeding the recent past as ordinary tabular features.

python · rolling_features.py
df['lag_1'] = df['sales'].shift(1)
df['lag_7'] = df['sales'].shift(7)
df['rolling_mean_7'] = df['sales'].shift(1).rolling(window=7).mean()
df['rolling_std_7'] = df['sales'].shift(1).rolling(window=7).std()
# .shift(1) before rolling prevents leaking the current day's value
Interview Q

Why can't you randomly shuffle-split time series data?

Tap to reveal
Answer

It would let the model train on future data to predict the past, a severe form of leakage. Time series validation must respect temporal order — using expanding or rolling-origin splits where training data always precedes the validation window.

Definition

What does the "I" in ARIMA stand for?

Tap to reveal
Answer

"Integrated" — the number of differencing operations ($d$) needed to transform a non-stationary series into a stationary one before fitting the AR/MA components.

Trade-off

ARIMA vs gradient-boosted trees for forecasting

Tap to reveal
Answer

ARIMA is statistically principled with interpretable parameters and works well on clean, single series with limited data. Tree-based models scale better to many series with rich exogenous features, but need manual lag/rolling feature engineering and offer less theoretical guarantee of unbiasedness.

Micro-Quiz

An ADF test on your sales series returns a p-value of 0.62. What should you do before fitting ARIMA?

p=0.62 is far above the typical 0.05 threshold, so you fail to reject $H_0$ (unit root present → non-stationary). Differencing the series (and re-running ADF) is the standard next step before fitting the AR/MA components.
Module 07

Decision Trees — Deep Dive

Every split a decision tree makes is the answer to one question: which feature and threshold most reduces impurity? Understanding the impurity math explains both the model's power and its tendency to overfit.

Splitting Criteria — The Math

For classification, two impurity measures dominate. Gini impurity measures the probability that two randomly drawn samples from a node have different classes:

$$Gini(t) = 1 - \sum_{k=1}^{K} p_k^2$$

Entropy measures information-theoretic disorder:

$$Entropy(t) = -\sum_{k=1}^K p_k \log_2(p_k)$$

A split is chosen to maximize information gain — the impurity reduction from parent to weighted children:

$$IG = I(parent) - \left(\frac{n_{left}}{n}I(left) + \frac{n_{right}}{n}I(right)\right)$$

For regression trees, splits minimize MSE (squared error from the node's mean prediction) or MAE (absolute error from the node's median prediction) within each resulting child node.

Bias-Variance & Pruning

An unconstrained tree grown to full depth has near-zero bias but very high variance — it memorizes training noise. Pruning trades a bit of training fit for generalization.

Cost Complexity Pruning (CCP) grows the full tree, then removes subtrees that don't justify their complexity, guided by a regularized cost:

$$R_\alpha(T) = R(T) + \alpha |T|$$

where $R(T)$ is the tree's training error, $|T|_i$ is the number of leaf nodes, and $\alpha \geq 0$ penalizes complexity. As $\alpha$ increases, larger subtrees get pruned; sklearn's ccp_alpha parameter exposes this directly, and is typically tuned via cross-validated cost-complexity paths.

python · pruned_tree.py
from sklearn.tree import DecisionTreeClassifier

# Pre-pruning via max_depth — stops growth at depth 5
tree = DecisionTreeClassifier(max_depth=5, random_state=42)
tree.fit(X_train, y_train)

# Post-pruning via cost-complexity path
path = DecisionTreeClassifier(random_state=42).cost_complexity_pruning_path(X_train, y_train)
ccp_alphas = path.ccp_alphas

# Cross-validate over alpha values to find the best generalizing subtree
best_tree = DecisionTreeClassifier(ccp_alpha=ccp_alphas[-5], random_state=42)

The tree greedily picks, at every node, the (feature, threshold) pair that maximizes this impurity reduction.

Trade-off

Gini vs Entropy in practice

Tap to reveal
Answer

They usually produce very similar trees. Gini is slightly cheaper (no logarithm) and is sklearn's default. Entropy is more sensitive to changes in class probability near the extremes, occasionally yielding marginally different splits.

Interview Q

Pre-pruning vs post-pruning

Tap to reveal
Answer

Pre-pruning (max_depth, min_samples_leaf) stops growth early — cheap but can miss useful splits hidden deeper. Post-pruning (CCP) grows the full tree first, then removes weak subtrees based on validated cost — generally yields better-generalizing trees at higher training cost.

Definition

What does $\alpha$ control in CCP?

Tap to reveal
Answer

The complexity penalty per leaf. $\alpha=0$ recovers the fully grown tree; increasing $\alpha$ makes additional leaves "cost" more, pruning away subtrees whose impurity reduction doesn't justify their added complexity.

Micro-Quiz

A decision tree achieves 100% training accuracy but only 68% test accuracy. What's the most direct fix?

100% train accuracy with a large train-test gap is the signature of an overfit, unconstrained tree memorizing training data. Constraining depth/leaf size or applying cost-complexity pruning directly reduces variance and closes that gap. Switching impurity criteria rarely fixes overfitting.
Module 08

Random Forests & Bagging Ensembles

A single tree is a high-variance estimator. Bagging trades a population of slightly-different, decorrelated trees for one stable, lower-variance prediction — and gets a free validation set as a byproduct.

Bootstrap Aggregation (Bagging)

Bagging trains $B$ trees, each on an independent bootstrap sample (sampling $n$ rows with replacement from the original $n$-row training set), then averages (regression) or majority-votes (classification) their predictions:

$$\hat{f}_{bag}(x) = \frac{1}{B}\sum_{b=1}^{B} \hat{f}_b(x)$$

Averaging $B$ independent, identically-distributed estimators with variance $\sigma^2$ reduces variance to $\sigma^2/B$ if they're truly independent. Trees from bagging alone are correlated (they see overlapping data), which caps the benefit — motivating Random Forest's extra decorrelation trick.

Random Forest: Feature Subspace Sampling

At each split, Random Forest restricts the candidate features to a random subset of size $m$ (typically $m = \sqrt{p}$ for classification, $m = p/3$ for regression) rather than all $p$ features. This decorrelates the trees: without it, a few dominant features would force every tree's top splits to look nearly identical, defeating the variance-reduction purpose of averaging.

Extra Trees (Extremely Randomized Trees) push this further by also randomizing the split threshold for each candidate feature (rather than searching for the optimal threshold), trading a bit of bias for even lower variance and faster training.

Out-of-Bag (OOB) Error — Validation for Free

Each bootstrap sample, drawn with replacement from $n$ rows, excludes roughly $1/e \approx 36.8\%$ of rows on average:

$$P(\text{row excluded}) = \left(1 - \frac{1}{n}\right)^n \xrightarrow{n\to\infty} e^{-1} \approx 0.368$$

These excluded ("out-of-bag") rows act as a free, built-in validation set for that specific tree. The OOB error averages each row's prediction error using only the trees that didn't see that row during training — giving an unbiased estimate of test error without sacrificing any data to a held-out split.

python · random_forest_oob.py
from sklearn.ensemble import RandomForestClassifier

rf = RandomForestClassifier(
    n_estimators=300,
    max_features='sqrt',
    oob_score=True,        # enables free validation via OOB samples
    n_jobs=-1,
    random_state=42
)
rf.fit(X_train, y_train)
print(f"OOB Score: {rf.oob_score_:.4f}")

Module 09

Gradient Boosting — XGBoost, LightGBM & CatBoost

Where bagging averages independent trees, boosting builds them sequentially, each one correcting the previous ensemble's mistakes. The modern GBM libraries differ mainly in how they grow trees and approximate the loss.

The Boosting Framework

Boosting builds an additive model stage-wise. At stage $m$, it adds a new weak learner $h_m$ that best reduces the current loss:

$$F_m(x) = F_{m-1}(x) + \eta \cdot h_m(x)$$

where $\eta$ is the learning rate (shrinkage) controlling how much each new tree is trusted. $h_m$ is fit not to the raw target but to the negative gradient of the loss with respect to the current predictions — i.e. each tree learns to predict the residual error direction of the ensemble so far.

XGBoost: Second-Order Taylor Approximation

XGBoost's key innovation is approximating the loss function around the current prediction using a second-order Taylor expansion, using both the gradient $g_i$ and Hessian $h_i$ of the loss:

$$\mathcal{L}^{(t)} \approx \sum_{i=1}^n \left[ g_i f_t(x_i) + \frac{1}{2}h_i f_t(x_i)^2 \right] + \Omega(f_t)$$

where $g_i = \partial_{\hat{y}} l(y_i, \hat{y}^{(t-1)})$, $h_i = \partial^2_{\hat{y}} l(y_i,\hat{y}^{(t-1)})$, and $\Omega(f_t) = \gamma T + \frac{1}{2}\lambda\sum_j w_j^2$ regularizes tree complexity ($T$ leaves, leaf weights $w_j$). For a fixed tree structure, the optimal leaf weight and the resulting quality score have closed forms:

$$w_j^* = -\frac{\sum_{i \in I_j} g_i}{\sum_{i \in I_j} h_i + \lambda}, \qquad \mathcal{L}^* = -\frac{1}{2}\sum_j \frac{(\sum_{i\in I_j} g_i)^2}{\sum_{i \in I_j} h_i + \lambda} + \gamma T$$

This closed-form score lets XGBoost evaluate candidate splits exactly, without needing to retrain — the basis for its exact and approximate split-finding algorithms.

Level-wise vs Leaf-wise Growth

XGBoost (level-wise / depth-wise) grows all nodes at the current depth before moving deeper, keeping the tree balanced. This is more conservative and resistant to overfitting on smaller datasets.

LightGBM (leaf-wise) always splits the single leaf with the highest loss reduction, regardless of depth — producing deeper, asymmetric trees that converge faster and often achieve lower loss for the same number of leaves, at higher overfitting risk on small data (mitigated via max_depth and min_data_in_leaf).

CatBoost: Native Categorical Handling

Naively target-encoding categories (using the full dataset's target mean) leaks information. CatBoost uses Ordered Target Statistics: for each row, it computes the category's target mean using only rows that appear before it in a random permutation, simulating an online/out-of-fold encoding without ever using a row's own label:

CatBoost also builds symmetric (oblivious) trees — every node at a given depth uses the same split condition — which makes prediction extremely fast and acts as a form of regularization.

Interview Q

Why does XGBoost need the Hessian, not just the gradient?

Tap to reveal
Answer

The second-order term lets XGBoost compute the exact optimal leaf weight and split-quality score in closed form for any candidate split, rather than relying only on a first-order (gradient-only) approximation — giving more precise splits and faster convergence.

Trade-off

LightGBM speed vs overfitting risk

Tap to reveal
Answer

Leaf-wise growth converges faster and often reaches lower training loss per leaf budget, but the resulting deep, unbalanced trees can overfit small datasets more easily than XGBoost's balanced level-wise trees — controlled via max_depth and min_data_in_leaf.

Definition

What is "Ordered Boosting" in CatBoost?

Tap to reveal
Answer

A technique where, for each row, the residual used to fit the next tree is computed using a model trained only on rows before it in a random permutation — avoiding the "target leakage" that occurs in standard boosting when residuals are computed using a model that already saw that row's label.

Micro-Quiz

You have a dataset with many high-cardinality categorical columns and limited time for feature engineering. Which library handles this most natively?

CatBoost was specifically designed to handle categorical features natively via ordered target statistics, avoiding both manual encoding work and the leakage that naive target encoding introduces — a major practical advantage for categorical-heavy tabular data.
